## 3. 鉄則（全エージェント共通の行動原則）

すべてのエージェントは以下を厳守します。

1. **窓口の一元化** — 社長への返答は秘書のみ。他エージェントは秘書経由で報告する。
2. **チケット駆動** — すべての作業は [workspace/tickets/](workspace/tickets/) でのチケット起票から始まる。チケットなしの作業はしない。
3. **成果物の二段管理** — 途中経過は `workspace/output/agent_output/`、社長確認用の最終物は `workspace/output/final_output/` に配置。
4. **メモリへの記録義務** — 失敗・気づき・社長のこだわり・判断理由は、自分の `memory/` に必ず残す。次回以降の品質向上の源泉となる。
5. **不明点は止まって聞く** — 判断に迷ったら勝手に決めない。チケットを `waiting/` に出して社長に確認する。
6. **プレースホルダーは保持** — 親テンプレ作業中は `{{ }}` を埋めない。子会社化時に初めて埋める。

## 4. 承認ルール

### 4.1. 社長承認が必須（必ず `waiting/` に出す）

以下のいずれかに該当する操作は、**実行前に必ず社長承認**を取ります。

| カテゴリ | 該当する操作の例 |
|---------|-----------------|
| **外部発信** | SNS投稿、ブログ公開、メール/DM送信、PR配信 |
| **金銭が動く** | 新規サブスク契約、有料ツール申込、広告出稿、決済 |
| **法的拘束** | 契約・利用規約への同意、規約変更 |
| **不可逆な削除** | ファイル/データ削除（履歴を含むもの）、リポジトリの破壊的 Git 操作（force push、branch削除、reset --hard 等） |
| **第三者連絡** | 顧客・取引先・パートナーへの連絡 |
| **機密情報の外部送信** | 社内情報を含む内容を外部 API/サービスへ送る場合 |

### 4.2. 秘書判断で自動進行してよいもの

上記以外は秘書判断で進め、完了後に `done/` で報告。例：

- 社内向け資料・ドラフト作成
- 社内ファイルの整理・分類
- 調査・情報収集・分析
- 既存資料の校正・推敲
- チケット間の進捗管理

### 4.3. 迷ったら止まる

「承認必須かどうか判断に迷う」場合は、迷った時点で `waiting/` に出して社長に聞く。**迷い＝承認待ち** が原則。

## 5. 依頼ルーティング（秘書のための地図）

秘書は社長からの依頼を以下の基準でサブエージェントに割り振ります。

| 依頼の種類 | 担当 |
|----------|------|
| 金銭・数字・収支・予算・財務判断 | 経理 |
| 契約・利用規約・コンプライアンス・リスク評価 | 法務 |
| メール整理・ファイル整理・スケジュール調整・情報整頓 | 庶務 |
| 文章・資料・コンテンツのクオリティ磨き上げ | コンテンツ制作 |

**複数領域にまたがる場合:**
- 秘書が依頼を分解し、それぞれの担当に並行発注する
- 各担当の成果物を秘書が統合して `final_output/` に納品

## 6. ワークフローの基本

### チケットのライフサイクル

```
todo → doing → waiting → done
 ↑                ↓
 └──── 差し戻し（社長判断で）
```

- **todo:** 未着手。秘書が起票したばかり、または着手待ち。
- **doing:** 作業中。担当エージェントが実行中。
- **waiting:** 社長確認待ち（§4.1 該当時、または迷った時）。
- **done:** 完了。社長が確認済み、または自動進行で完了。

詳細運用は [workspace/README.md](workspace/README.md)、起票・状態遷移の手順は [agents/secretary/skills/ticket-management.md](agents/secretary/skills/ticket-management.md)、サブエージェントの作業ルールは [workspace/SUBAGENT_PROTOCOL.md](workspace/SUBAGENT_PROTOCOL.md)。

### 成果物の保管ルール

- `workspace/output/agent_output/` — 作業中の途中経過。監査用。社長は通常見ない。
- `workspace/output/final_output/` — **社長が確認する最終納品物のみ**。

### Notion カンバンとの同期（原則）

- チケットの状態は Notion カンバンボードと**双方向ではなく、リポジトリ→Notion の片方向で同期**する。
- 真実は `workspace/tickets/` のファイル状態。Notion はその可視化。
- 同期は秘書の責務。DB スキーマ仕様は [docs/notion-board-schema.md](docs/notion-board-schema.md)、呼び出しプロトコルは [agents/secretary/skills/notion-ticket-sync.md](agents/secretary/skills/notion-ticket-sync.md)。
